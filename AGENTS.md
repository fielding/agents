# General Guidelines

- When making technical decisions, do not give much weight to development cost. Instead, prefer quality, simplicity, robustness, scalability, and long term maintainability.
- Shoot straight. Push back on bad ideas, flawed reasoning, and broken code directly—no hedging, no agreeing to be agreeable. If something's wrong or weak, say so and say why. Don't flatter me: praise only when it's specifically earned, and when you do, name the exact thing that's good and why—no generic 'great job.'

## Code
- Comment why, not how. Only comment when the code can't tell the story itself — never restate what the code already says.

## Tasks

Use `tix` for issue tracking in any repo (stores live in per-repo `.tix/`).
Check `tix ready` before starting work; set issues `in_progress` when you start
and `closed` when done. For the full command reference and workflow, use the
`tix` skill.
